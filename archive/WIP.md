# Consolidated Voca Agent Development Plan

This document outlines the consolidated development plan for the Voca agent, merging the strategic direction from `NEWPLAN.md` with the ongoing work and learnings previously tracked in `WIP.md`. The goal is to create a cohesive roadmap for enhancing the agent's capabilities, user experience, and reliability.

## Key Learnings & Context (From Recent Investigation)

*   **Gemini API Response Structure (`parts`):** The API response (`candidates[0].content.parts`) can contain multiple distinct parts, such as `text` and `functionCall`, within a single turn. Our current issue stems from the LLM generating *only* the `<llm_scratchpad_update>` within its `text` part when also generating a `functionCall`, omitting the required user-facing conversational text.
*   **Structured JSON Output (`responseSchema`):** Gemini officially supports constraining the LLM's textual output to conform to a specific JSON schema provided via the `responseSchema` field in `generationConfig`. This is distinct from function calling and offers a robust way to structure the model's "thinking" output.
*   **Function Calling vs. `responseSchema`:** Function calling structures the *intent to call a function and its arguments*. `responseSchema` structures the *textual response* generated by the model. For our goal, we need the LLM to successfully produce *both* a `functionCall` part *and* a `text` part that adheres to our `responseSchema` within a single API response.
*   **Multi-Part Reliability:** While possible, getting the LLM to reliably generate multiple distinct components (user-facing text adhering to schema, scratchpad update adhering to schema, *and* a function call) simultaneously in a complex prompt scenario might be challenging due to potential instruction overload or prioritization conflicts.
*   **Voca vs. Standard Chat Flow:** Voca's requirement to bundle user-facing confirmation text *with* the `stage_add` function call (to optimize turns) is a more complex interaction pattern than a typical chat flow where explanation/action/confirmation might span multiple turns.
*   **Prompting Challenges:** The current prompt's length, multiple "MUST" instructions (especially for scratchpad updates), and the order of instructions could be contributing factors to the LLM dropping the user-facing text requirement when also handling a function call and scratchpad update.
*   **Existing Fallback:** The logic for making a separate, dedicated LLM call for confirmation messages when the primary response lacks text *is already partially implemented* in `src/loop.ts` as a fallback mechanism.
*   **Staging Location:** `staged_changes` are correctly part of the persisted `AgentState` (in Redis), not just the scratchpad.

## E2E Test Flakiness Investigation Summary (apply_staged_changes - delete operation)

*   **Core Problem:** The E2E test `it('should stage create, apply, stage delete, apply, then verify deletion')` in `src/tests/loop/apply_staged_changes.e2e.spec.ts` was intermittently failing. The `staged_changes` array in `AgentState` was not empty after the delete operation was supposedly applied, indicating the deletion change wasn't removed from state.
*   **Key Discovery (State Overwriting):** The save operation from the "apply create" turn was completing *after* the E2E test had staged the new delete operation for the subsequent turn, but *before* `runConversationTurn` for the "apply delete" turn could load the state.
*   **The Fix:** The `await saveAgentState(...)` call *within* `apply_staged_changes` (in `src/tools/linear_apply.ts`) was commented out. This made `runConversationTurn` the sole function responsible for saving the `AgentState` at the end of a complete turn.
*   **Implication:** This experience underscored the need for a more robust state management strategy, leading to Phase 1.4.

## Phased Action Plan

**Phase 1: Implement LLM-Flavored, App-Delivered Tool Call Progress Messages**

**Goal:** Enhance UX by having the LLM generate contextual "pre-tool" and "post-tool-completion" messages, delivered by the application, while ensuring the core `functionCall` mechanism remains robust. This phase also aims to directly address and fix the current E2E test failures ("Empty Agent Responses" and "LLM Re-requests Tool") by ensuring the LLM reliably outputs a `functionCall` when a tool is intended.

**Part 1.A: Stabilizing the Core `functionCall` Output (Fixing E2E Tests & Summarizing Tool Outputs)**

*   **Problem (Original from NEWPLAN.md):** The LLM sometimes produces *only* a text response (containing the JSON for `userFacingMessage` and `scratchpadContent`) without the `functionCall` object, even when a tool is clearly intended by the user's query and the LLM's `userFacingMessage`.
*   **Problem (Integrated from WIP.md - Old Phase 2.5 - CRITICAL CURRENT FOCUS):** The LLM frequently fails to generate a user-facing message summarizing the results after a tool (e.g., `linear_search`, `linear_get_details`) has executed. Instead, it may produce an empty response or attempt to re-call the same tool. This is a primary blocker for reliable agent operation.


*   **Hypothesis (from NEWPLAN.md):** The current `GEMINI_TASK_SPECIFIC_INSTRUCTIONS` and the way the LLM is asked to structure its `text` response (the JSON with `userFacingMessage` and `scratchpadContent`) might be confusing it or leading it to believe that providing this structured text *is* the primary task, overshadowing the need to *also* emit a `functionCall` object when a tool is necessary.
*   **Underlying Cause for Failure to Summarize (Hypothesis from WIP.md):**
    1.  Lack of explicit system prompt instructions guiding the LLM on how to behave *after* a tool result is present in the history.
    2.  The second LLM call in `runConversationTurn` (after tool execution) receives an empty `systemInstruction` and an empty `newMessage`, providing no immediate, specific guidance for processing the tool\'s output.


*   **Strategy (Combined and Prioritized):**
    1.  **Simplify LLM's First Turn Output Expectation (When a Tool is Called - from NEWPLAN.md):**
        *   Modify the prompting strategy. When the LLM decides to call a tool, its *primary* output should be the `functionCall` object.
        *   The `userFacingMessage` and `scratchpadContent` will now be handled differently:
            *   The "pre-execution narration" (see Part 1.B) will serve as the initial `userFacingMessage`.
            *   The `scratchpadContent` update will happen after the *second* LLM call (the synthesis call), or if no tool is called, after the first LLM call.
    2.  **Refine `GEMINI_TASK_SPECIFIC_INSTRUCTIONS` and `GEMINI_SYSTEM_PROMPT_STATIC_PARTS` (from NEWPLAN.md):**
        *   Remove instructions that demand the LLM output a JSON string containing `userFacingMessage` and `scratchpadContent` *in the text part of the first response if a tool is also being called*.
        *   **Emphasize:** "If a tool is needed to fulfill the user\'s request, your *sole* output related to that tool in this turn should be the `functionCall` object. You will have a chance to provide a user-facing message and update your scratchpad *after* the tool runs, or if no tool is called."
    3.  **Tool Invocation Logic (`src/loop.ts` - `runConversationTurn` - from NEWPLAN.md):**
        *   The logic for `if (firstFunctionCalls && firstFunctionCalls.length > 0)` remains central.
        *   If `firstFunctionCalls` is present, proceed with tool execution.
        *   If `firstFunctionCalls` is *not* present, then the `rawLlmTextOutput` from the first LLM call is considered the final response. This text should now be a direct user-facing message.
    4.  **Implement Targeted System Instruction for Post-Tool Summarization (Top Priority from WIP.md - Old Phase 2.5.A):**
        *   **File:** `src/loop.ts` (within `runConversationTurn`, for the second LLM call).
        *   **Action:** Create and pass a new, specific `systemInstruction` string to `geminiClient.sendMessageStream` for the LLM call that occurs *after* a tool has executed. This instruction will explicitly direct the LLM to:
            1.  Analyze the most recent tool result from the conversation history.
            2.  Formulate a concise, user-facing message summarizing these results or informing the user of the outcome (as `userFacingMessage`).
            3.  Update its scratchpad (`scratchpadContent`).
            4.  Avoid re-calling tools unless essential for a multi-step process and the user hasn\'t yet responded to the previous tool\'s output.
            5.  Ensure the entire response is a single JSON object adhering to `StructuredLlmResponse`.
        *   A new prompt constant will be created (e.g., in `src/prompts/system_post_tool_instructions.ts`).
    5.  **Enhance General Task-Specific Instructions for Summarization (Supporting from WIP.md - Old Phase 2.5.B):**
        *   **File:** `src/prompts/system_task_specific_instructions.ts`.
        *   **Action:** Add/modify instructions to generally reinforce the expectation that when tool results (from `linear_search`, `linear_get_details`, etc.) appear in the history, the LLM\'s next step is to summarize them for the user and await further input, rather than re-calling tools.
    6.  **Addressing "LLM Re-requests Tool" in the Second LLM Call (from NEWPLAN.md - aligns with summarization goal):**
        *   In `runConversationTurn`, when preparing `historyForSecondLlmCall`:
            *   **Crucially, do NOT pass `availableTools` to `geminiClient.generateContentWithTools` for the second LLM call.** The second call's sole purpose is to synthesize the tool's result into a user-facing message and update the scratchpad.
            *   `const secondLlmResponse = await geminiClient.generateContent(historyForSecondLlmCall);` (or equivalent, pass empty tools array or `tool_config: {mode: "NONE"}`).
    7.  **Provide `newMessage` for Second LLM Call (If Necessary - from WIP.md - Old Phase 2.5.C):**
        *   **File:** `src/loop.ts`.
        *   **Action:** If prior actions are insufficient, consider passing a non-empty `newMessage` (e.g., "Please process the latest tool result and respond to the user.") to the second LLM call.

*   **Success Criteria for Part 1.A (Combined):**
    *   **Primary (from WIP.md):** Significant improvement in E2E test pass rates, specifically:
        *   `src/tests/e2e/loop/basic_loop.e2e.spec.ts` (must pass)
        *   `src/tests/e2e/loop/linear_search.e2e.spec.ts` (must pass)
        *   Expected positive impact on other `linear_*` and `stage_*` E2E tests.
    *   LLM reliably outputs `functionCall` when a tool is intended.
    *   LLM reliably summarizes tool outputs in a subsequent turn instead of re-requesting or providing empty responses.
    *   **Qualitative (from WIP.md):** When observing agent interactions, after a tool successfully executes and returns data, the agent\'s subsequent textual response to the user clearly and correctly summarizes that data or informs of the outcome. Unnecessary tool re-invocations for the same underlying request cease.

**Part 1.B: Implementing Narration Parameters**

1.  **Modify Tool Schemas (`src/tools/`):**
    *   For each relevant `GeminiFunctionDeclaration` (e.g., `linearSearchToolSchema`, `linearGetDetailsToolSchema`, `stageAddToolSchema`, etc.):
        *   Add one new **optional** string property to their `parameters.properties`:
            *   `pre_execution_narration`:
                *   `type: Type.STRING`
                *   `description: "Optional. A short, friendly message (1-2 sentences) to display to the user *before* this tool executes, briefly explaining what you are about to do with this tool. Infuse your current persona. Example: 'Alright, I'll search for those bugs now!' or 'Let me fetch the details for that ticket.'"`
    *   This parameter should NOT be in the `required` array of the tool schemas.

2.  **Update Prompting Strategy (e.g., in `GEMINI_TASK_SPECIFIC_INSTRUCTIONS` or tool descriptions):**
    *   Instruct the LLM: "When you decide to call a tool, if you wish to provide a brief narration, populate the `pre_execution_narration` parameter within the tool's arguments. This should be short and engaging. If you populate `pre_execution_narration`, it will be shown to the user before the tool runs."

3.  **Modify `src/loop.ts` - `runConversationTurn`:**
    *   **Inside `runConversationTurn`, if `firstFunctionCalls` is present:**
        *   Before calling `executeToolCall`, extract `pre_execution_narration` from `currentToolCall.args`.
        *   If `pre_execution_narration` is present and non-empty:
            *   Add it to `agentState.conversation_history` with `role: 'assistant'`.
            *   **Crucially, send this message to the user interface immediately.** (This might require a mechanism to send partial/intermediate responses to the frontend if not already present).
    *   **Inside `executeToolCall` (or right after it returns in `runConversationTurn`):**
        *   After the actual tool function (e.g., `linear_search_issues_tool(...)`) completes (successfully or with an error string):
            *   Extract `post_execution_acknowledgement` from `currentToolCall.args` (the `currentToolCall` object needs to be available here or its `args` passed down).
            *   If `post_execution_acknowledgement` is present and non-empty:
                *   Add it to `agentState.conversation_history` with `role: 'assistant'`.
                *   **Send this message to the user interface immediately.**

4.  **Scratchpad Update Logic:**
    *   The `llm_scratchpad` should be updated:
        *   If no tool is called: After the first LLM call, based on its direct textual response (which might contain implicit instructions or be parsable for scratchpad content if you design a new, simpler convention).
        *   If a tool is called: After the *second* LLM call (the synthesis call), based on *its* textual response. This keeps the first tool-calling turn cleaner.
        *   You might need to adjust `extractLlmResponseAndScratchpad` or how it's used. The complex JSON parsing for `userFacingMessage` + `scratchpadContent` should be removed or simplified for the first LLM call when a tool is also present.

**Phase 2: Implement LLM-Managed Plan & Execution (Iterative)**

**Goal:** Enable Voca to handle multi-step user requests by having the LLM generate, manage, and update a plan stored in `AgentState.llm_scratchpad` (or a new dedicated `AgentState.current_plan` field).

1.  **Define the `Plan` Schema:**
    *   Design a JSON schema for the `Plan` object (similar to the example in our previous discussion, including `original_user_goal`, `overall_status`, `steps` array, `next_step_id_to_execute`, `reasoning_log`, `final_answer_to_user`).
    *   Each `step` should have `id`, `description` (LLM-generated for clarity), `tool_to_call`, `tool_arguments` (potentially with placeholders like `${step_1.result.some_value}`), `status`, `result_summary`, `tool_output_reference`.

2.  **New "Planner/Manager" System Prompt Section:**
    *   Add a new major section to your system prompt instructions (e.g., to `GEMINI_SYSTEM_PROMPT_STATIC_PARTS` or a new dedicated prompt segment).
    *   **Instructions for the LLM:**
        *   "For complex requests that require multiple steps or tools, you will act as a planner and manager."
        *   "First, formulate a `Plan` object matching this schema: [describe schema]. Store this plan in your `llm_scratchpad` (or a dedicated field if you create one)."
        *   "In each turn, review the current `Plan` from your scratchpad."
        *   "Based on the plan, the `next_step_id_to_execute`, and the result of any previously executed tool for this plan, decide your next action:"
            *   "**If a tool needs to be called for the next step:** Output a `functionCall` for that tool. You *must* populate `pre_execution_narration` if you want to narrate this step."
            *   "**If the plan is complete:** Generate your `final_answer_to_user` as a direct textual response. Update the plan's `overall_status` to 'completed'."
            *   "**If you need clarification from the user:** Ask your question as a direct textual response. Update the plan's `overall_status` to 'clarification_needed'."
            *   "**After deciding your action, update the `Plan` object in your scratchpad:** Reflect the status of the executed step, add to `reasoning_log`, update `next_step_id_to_execute`, and potentially modify future steps if re-planning is needed."
            *   "**Safety First:** For any step involving data modification (e.g., using `stage_add_issue`, `apply_staged_changes`), the plan *must* first include a step where you ask the user for explicit confirmation via a textual message. Only proceed with the modification tool call after user confirmation."
        *   "Your textual response (if not making a `functionCall`) should clearly indicate if you are providing a final answer, asking for clarification, or confirming an action. Always update your `llm_scratchpad` with the latest version of the `Plan`."

3.  **Modify `src/loop.ts` - `runConversationTurn` (Orchestrator Logic):**
    *   The loop will become more stateful regarding the plan.
    *   **Initial Detection of a Multi-Step Request (Heuristic or LLM Signal):**
        *   How does Voca know a request needs a "plan" vs. a single tool call?
            *   Option 1 (LLM decides): The LLM, based on the user query and the new planning instructions, might decide to initiate a plan itself by starting to populate the plan structure in its scratchpad.
            *   Option 2 (Explicit Trigger): You could have a specific way for the user to invoke "planning mode" or your application might use heuristics. For now, let the LLM decide.
    *   **Main Loop Iteration (when a plan is active):**
        1.  LLM receives history including the current `Plan` from `llm_scratchpad`.
        2.  LLM outputs either:
            *   A `functionCall` for the `next_step_id_to_execute` (with optional narrations).
            *   A textual response (final answer, clarification, confirmation question).
            *   Crucially, its textual response (or a convention within it) should also contain the *updated Plan object* intended for the scratchpad.
        3.  **Application Logic:**
            *   Parse the LLM's output. Update `agentState.llm_scratchpad` with the new `Plan`.
            *   If `functionCall`:
                *   Extract and display `pre_execution_narration`.
                *   Execute the tool specified in the plan step.
                *   Display `post_execution_acknowledgement`.
                *   The tool's result (data or error string) is prepared. This result needs to be clearly associated with the step that produced it, perhaps by adding it to a temporary part of `AgentState` or directly into the conversation history in a structured way that the LLM can reference in the next turn (e.g., "Result of plan step `step_1`: [...]").
                *   Loop back to let the LLM process this result and the updated plan.
            *   If textual response:
                *   Send to the user.
                *   If `plan.overall_status` is 'completed' or 'failed', the planning session for this goal might end.
                *   If 'clarification_needed' or waiting for confirmation, await next user input.
    *   **Argument Referencing:** The LLM needs to be able to use outputs from previous plan steps as inputs to later steps (e.g., `tool_arguments: { "issue_id": "${step_1.result.most_critical_issue_id}" }`). Your application will need to:
        *   Store tool outputs in a way that can be referenced (e.g., `AgentState.plan_step_outputs['step_1'] = { most_critical_issue_id: '...' }`).
        *   Before calling a tool for a plan step, resolve these placeholders in `tool_arguments` using the stored outputs.

4.  **`AgentState` Modifications:**
    *   The `llm_scratchpad` will now be more strictly managed by the LLM to hold the `Plan` object.
    *   You might consider adding `current_plan: Plan | null` and `plan_step_outputs: Record<string, any>` directly to `AgentState` for easier access and less reliance on parsing the string scratchpad if it becomes too complex. For now, using the scratchpad is fine.

**Refactoring Core Loop for Structured Agent Output (`AgentTurnOutput`)**

**Goal:** To refactor `src/loop.ts` so that its main function `runConversationTurn` (and consequently, how the agent communicates its results) returns a well-defined, structured object (`AgentTurnOutput`). This will make the agent's state, actions, and intended communications explicit, significantly improving the system's testability, maintainability, and the ability for consuming interfaces (like a UI or more robust E2E tests) to react intelligently to the agent's state. This refactor directly supports and enhances the reliability of both Phase 1 (Narration & Summarization) and Phase 2 (Planner).

**Context & Motivation:** Previous attempts to modify `src/loop.ts` for this purpose highlighted the complexity and the need for a clear, phased approach. The current effort restarts this specific refactoring with a more granular plan. The type definitions in `src/types/agent-output.ts` are already in place from a prior attempt.

**Phases for `AgentTurnOutput` Implementation:**

*   **Phase A: Foundational Type Definition Review (Completed & Verified)**
    *   **Action:** Ensure `AgentTurnOutput` and its constituent types (`AgentIntent`, `PendingAction`, `ToolExecutionResult`, etc.) as defined in `src/types/agent-output.ts` are robust and cover necessary scenarios.
    *   **Status:** Considered complete. The types were defined in a previous session.

*   **Phase B: Core Function Signature & Basic Return Path Adaptation**
    *   **Goal:** Modify `runConversationTurn` in `src/loop.ts` to adopt the new `Promise<AgentTurnOutput | undefined>` return signature and update its most straightforward return paths.
    *   **Actions:**
        1.  Change the return signature of `runConversationTurn`.
        2.  Modify direct error returns (e.g., agent state not found) to wrap messages in the `AgentTurnOutput` structure, using appropriate `AgentIntent` like `'AGENT_ERROR'` and populating `internalErrorDetails`.
        3.  Adapt simple text-only response paths to return `AgentTurnOutput`, setting `textResponse` and a placeholder `intent` (e.g., `'INFORMATION_PROVIDED'`).
        4.  Ensure the import for `AgentTurnOutput` and related types from `../types/agent-output` is correct in `src/loop.ts`. Resolve any import path issues (a previous blocker).
    *   **Focus:** Achieve a compilable state with the new signature and basic structured returns, minimizing disruption to complex internal logic initially.

*   **Phase C: Adapting Tool Call and Processing Logic**
    *   **Goal:** Integrate `AgentTurnOutput` population throughout the tool calling, execution, and result summarization lifecycle within `runConversationTurn` and its helper `runToolAndGenerateResponse`.
    *   **Actions:**
        1.  When a tool call is initiated by the LLM:
            *   Determine and set `AgentTurnOutput.intent` (e.g., `'TOOL_CALLED'`, or directly to `'AWAITING_CONFIRMATION'` if the tool implies it).
            *   Populate `AgentTurnOutput.pendingAction` with details of the tool about to be called (e.g., `type: 'CALL_TOOL'`, `toolName`).
            *   Ensure `textResponse` includes any pre-execution narration.
        2.  After a tool successfully executes and its results are summarized by the LLM (second LLM call):
            *   Set `AgentTurnOutput.intent` (e.g., `'ACTION_COMPLETED'`, `'INFORMATION_PROVIDED'`).
            *   Populate `AgentTurnOutput.toolResult` with `toolName`, `resultSummary`.
            *   Clear or update `pendingAction`.
        3.  If a tool execution fails:
            *   Set `AgentTurnOutput.intent` to `'TOOL_ERROR'`.
            *   Populate `AgentTurnOutput.toolResult` with `toolName` and the `error` message.
            *   Ensure `textResponse` communicates the error to the user.
            *   Populate `internalErrorDetails` if appropriate.
